Executives:
- GET /execs: Get list of executives
- 'POST /execs': Add a new executive
- "PATCH /execs': Modify multiple executives
- 'GET /execs/{id}": Get a specific executive
- "PATCH /execs/{id}': Modify a specific executive
- 'DELETE /execs/{id}": Delete a specific executive
- "POST /execs/login': Login
- POST /execs/logout: Logout
- "POST /execs/forgotpassword': Forgot password
- "POST /execs/resetpassword/reset/{resetcode}': Reset password

// BEST practices
MOdularity - Keep your API endpoints modular and independent
Each endpoint should have single responsibility
Document each endpoint with clear instructions on how to use it.
Expected inputs and possible outputs.
Implement a comprehensive error handling to provide meaningful error messages and HTTP status codes and always validate and sanitize inputs to prevent security vulnerabilities such as SQL injection and cross site scripting attacks.
Finally, write unit and integration tests to ensure the API works as expected and handles edge cases gracefully

// Common Pitfall
Overcomplicating the API
Avoid adding unnecessary complexity which can make API hard to maintain.
Always prioritize security from the beginning.
It is much harder to add security features after the API is built.
Test Thoroughly
Document properly
